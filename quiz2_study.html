Slow start → congestion avoidance: cwnd tracing
Q34 type · 4 marks
▶
How it works

TCP starts in Slow Start: cwnd doubles each RTT. When cwnd hits ssthresh, it switches to Congestion Avoidance: +1 MSS per RTT. On RTO (timeout): reset cwnd=1, new ssthresh = old cwnd/2.

-- Slow Start (cwnd < ssthresh) -- cwnd(n+1) = cwnd(n) × 2 // doubles each RTT -- Congestion Avoidance (cwnd ≥ ssthresh) -- cwnd(n+1) = cwnd(n) + 1 MSS // +1 per RTT -- On RTO -- ssthresh = max(cwnd/2, 2·MSS) cwnd = 1 MSS // restart slow start
Exact Q2 PYQ — Q34 worked solution
Given: cwnd=1 MSS, MSS=2KB, ssthresh=32KB. No loss for 4 RTTs. After SS ends at 32KB, 3 more RTTs of CA. Find final cwnd in KB.
SS starts: cwnd = 1 MSS = 2 KB
RTT1: 2×2 = 4 KB   RTT2: 4×2 = 8 KB
RTT3: 8×2 = 16 KB   RTT4: 16×2 = 32 KB = ssthresh → enter CA
CA RTT5: 32+2 = 34 KB
CA RTT6: 34+2 = 36 KB
CA RTT7: 36+2 = 38 KB
Answer: 38 KB ✓
⚠ Trap
When cwnd reaches ssthresh it does NOT double again. It adds exactly 1 MSS in CA. Also: count RTTs carefully — the problem said "4 RTTs no loss, then slow start ENDS at 32KB, then 3 more RTTs". The 4 RTTs take you TO ssthresh, not past it.
Practice — attempt before revealing
A TCP connection starts with cwnd = 1 MSS = 1 KB, ssthresh = 16 KB. There is no packet loss for 6 RTTs. What is the cwnd (in KB) after those 6 RTTs?
Worked solution:
SS: RTT1=2, RTT2=4, RTT3=8, RTT4=16KB → hits ssthresh, enter CA
CA: RTT5=16+1=17KB, RTT6=17+1=18KB... wait — MSS=1KB so +1KB per RTT
RTT5: 17 KB, RTT6: 18 KB... hmm, but the question says 6 RTTs total.
RTT1:2, RTT2:4, RTT3:8, RTT4:16(=ssthresh, enter CA), RTT5:17, RTT6:18, but that's 6 RTTs and answer = 18 KB.

Wait — at RTT4 cwnd reaches 16=ssthresh. Does RTT4 end at 16 (entering CA) or does SS go to 16 and then CA adds? Entering CA means next RTT uses +1. So RTT5=17, RTT6=18. Answer: 18 KB

cwnd in bytes after N ACKs in congestion avoidance
Q31 type · 4 marks
▶
The formula

In CA, each individual ACK increments cwnd by a fraction. After one full RTT (cwnd many ACKs), the net gain is exactly 1 MSS.

-- Per-ACK increment in CA -- Δcwnd_per_ack = MSS² / cwnd_bytes -- After one RTT (cwnd ACKs received) -- new_cwnd = cwnd + 1 MSS // net = 1 MSS per RTT new_cwnd_bytes = (cwnd_MSS + 1) × MSS_bytes
Exact Q2 PYQ — Q31 worked solution
cwnd = 15 MSS, MSS = 1200 bytes, in CA, 15 ACKs arrive (1 RTT worth). Find new cwnd in bytes.
Each ACK: Δ = 1200² / (15×1200) = 1200/15 = 80 bytes
15 ACKs: total gain = 15 × 80 = 1200 bytes = 1 MSS
new cwnd = (15+1) × 1200 = 16 × 1200
Answer: 19200 bytes ✓
⚠ Trap
Do NOT just add 1200. You must convert to bytes: (cwnd_MSS + 1) × MSS_bytes. The number of ACKs in 1 RTT = cwnd (in MSS).
Practice
cwnd = 20 MSS, MSS = 500 bytes, TCP is in congestion avoidance. After receiving 20 ACKs (one RTT), what is the new cwnd in bytes?
After 1 RTT in CA: new cwnd = (20+1) × 500 = 21 × 500 = 10500 bytes

TCP throughput from steady-state cwnd
Q32 type · 4 marks
▶
throughput = (cwnd × MSS_bytes) / RTT_seconds -- Unit conversions -- RTT: ms → s (÷ 1000) Result in B/s → Mbps (÷ 10⁶) Result in B/s → KB/s (÷ 1024)
Exact Q2 PYQ — Q32 worked solution
cwnd = 50 MSS, MSS = 1200 bytes, RTT = 80 ms. Find throughput in Mbps.
Data per RTT = 50 × 1200 = 60,000 bytes
RTT = 80 ms = 0.080 s
Throughput = 60,000 / 0.080 = 750,000 B/s
= 750,000 / 10⁶ Mbps = 0.75 Mbps... wait
= 750,000 × 8 bits/s = 6,000,000 bps = 6 Mbps
Answer: 6.00 Mbps ✓ (range: 5.90–6.10)
⚠ Trap
The formula gives bytes/second. To convert to Mbps multiply by 8 (bits per byte) then divide by 10⁶. Do NOT just divide B/s by 10⁶ — that gives Megabytes/s not Megabits/s.
Practice
A TCP connection has cwnd = 40 MSS, MSS = 1500 bytes, RTT = 100 ms. What is the maximum achievable throughput in Mbps?
Data = 40×1500 = 60,000 B/RTT
Throughput = 60,000/0.1 = 600,000 B/s
= 600,000 × 8 / 10⁶ = 4.8 Mbps

Jain's Fairness Index
Q33 type · 4 marks
▶
JFI = (Σxᵢ)² / (n × Σxᵢ²) -- Where -- xᵢ = bandwidth of flow i n = number of flows Range: 1/n (most unfair) to 1.0 (perfect fairness)
Exact Q2 PYQ — Q33 worked solution
3 flows: x₁=50 Mbps, x₂=30 Mbps, x₃=40 Mbps. Find JFI.
Σxᵢ = 50+30+40 = 120
(Σxᵢ)² = 120² = 14,400
Σxᵢ² = 50²+30²+40² = 2500+900+1600 = 5000
n × Σxᵢ² = 3 × 5000 = 15,000
JFI = 14,400 / 15,000 = 0.96
Answer: 0.96 ✓ (range: 0.94–0.98)
Perfect fairness: all flows equal → JFI = 1.0. If one flow hogs all bandwidth → JFI = 1/n. UDP flows don't back off → can reduce JFI of TCP flows.
Practice
Two TCP flows share a 100 Mbps link: Flow 1 gets 60 Mbps, Flow 2 gets 40 Mbps. What is the Jain's Fairness Index? (2 decimal places)
Σxᵢ = 100, (Σxᵢ)² = 10,000
Σxᵢ² = 3600+1600 = 5200
n×Σxᵢ² = 2×5200 = 10,400
JFI = 10,000/10,400 = 0.96

Fast retransmit & fast recovery cwnd
Q24 + Q37 type · 3+5 marks
▶
-- On triple duplicate ACK (fast retransmit) -- ssthresh = inflight / 2 cwnd = ssthresh + 3×MSS // RFC 2581 -- During fast recovery (each dup ACK) -- cwnd += 1 MSS // inflates cwnd -- When new ACK arrives (fast recovery ends) -- cwnd = ssthresh // deflate back to ssthresh // then continue with additive increase
Exact Q2 PYQ — Q37 worked solution
MSS=2KB, inflight=80KB (just before triple dup ACK). Find cwnd immediately after fast retransmit (cwnd = ssthresh + 3×MSS).
ssthresh = inflight/2 = 80/2 = 40 KB
cwnd = ssthresh + 3×MSS = 40 + 3×2 = 40+6 = 46 KB
Answer: 46 KB ✓
Exact Q2 PYQ — Q24 worked solution
During fast recovery: cwnd=8 MSS, inflight=4 MSS. 4 dup ACKs arrive. Find cwnd just BEFORE new ACK, and AFTER new ACK (when cwnd and inflight realign).
ssthresh was set = inflight_at_loss/2. Since cwnd starts at ssthresh+3, and cwnd=8, ssthresh=5... but given inflight=4 → ssthresh = 4/2 = 2... the question gives the state DURING recovery.
Each dup ACK: cwnd += 1 MSS
After 4 dup ACKs: cwnd = 8+4 = 12 MSS (just before new ACK)
New ACK arrives → cwnd deflates to ssthresh
ssthresh from the original loss = inflight/2 = 8/2 = 4 MSS (original cwnd=inflight=8)
Answer: 12 MSS before new ACK, 4 MSS after ✓ (Option C)
⚠ Trap — Q24 key insight
The question says "cwnd=8, inflight=4" at the START of this observation. This means the fast retransmit ALREADY happened and brought inflight to 4. The original cwnd at time of loss was 8 → ssthresh=8/2=4. During recovery cwnd inflates (+1 per dup ACK). On new ACK it deflates back to ssthresh=4. Remember: ssthresh is set ONCE at the start of fast retransmit, not re-calculated from current inflight.
Practice
A TCP flow has MSS = 1 KB. Just before a triple dup ACK, cwnd = inflight = 60 KB. What is cwnd immediately after fast retransmit (cwnd = ssthresh + 3×MSS), in KB?
ssthresh = 60/2 = 30 KB
cwnd = 30 + 3×1 = 33 KB

CIDR: usable host count from prefix
Q28 type · 3 marks
▶
usable_hosts = 2^(32 − prefix) − 2 -- Subtract 2 for: network address + broadcast address -- /26 → 2^6 − 2 = 64 − 2 = 62 /24 → 2^8 − 2 = 254 /30 → 2^2 − 2 = 2 // minimum usable
Q28: Block = 10.25.4.0/26. How many usable hosts?
host bits h = 32−26 = 6
usable = 2^6 − 2 = 64 − 2 = 62
Answer: 62 ✓
Practice
How many usable host addresses are in the block 192.168.10.0/28?
h = 32−28 = 4 → 2^4−2 = 16−2 = 14
2
Find smallest prefix to support N hosts
Q30 type · 3 marks
▶
-- You need: 2^h − 2 ≥ N -- h = ⌈log₂(N + 2)⌉ prefix = 32 − h -- Memory trick -- Round N up to next power of 2, then add 2 buffer. For N=500: need 2^h ≥ 502 → h=9 (2^9=512) → prefix=23
Q30: Need ≥500 hosts per subnet. Smallest prefix?
2^h − 2 ≥ 500 → 2^h ≥ 502
2^8 = 256 (too small), 2^9 = 512 ≥ 502 ✓
h = 9, prefix = 32−9 = 23
Answer: /23 ✓
⚠ Trap
Always test 2^h − 2 ≥ N (not just 2^h ≥ N). For N=500: 2^9−2=510 ≥ 500 ✓. If N=510: 2^9−2=510 ✓ still /23. If N=511: 2^9−2=510 < 511, need h=10 → /22.
Practice
An admin needs subnets that each support at least 1000 hosts. What is the smallest CIDR prefix length they can use?
2^h − 2 ≥ 1000 → 2^h ≥ 1002
2^10=1024 ≥ 1002 ✓, h=10
prefix = 32−10 = /22

Network address: bitwise AND with netmask
Q35 type · 4 marks
▶
network_address = IP_address AND netmask // Apply byte by byte in binary or use shortcut -- Shortcut rules -- byte AND 255 = byte unchanged byte AND 0 = 0 byte AND 240 = zero out lower 4 bits // 240 = 11110000 byte AND 128 = zero out lower 7 bits // 128 = 10000000
Q35: IP=172.25.50.22, Mask=255.255.240.0. Network address of A?
Byte 1: 172 AND 255 = 172
Byte 2: 25 AND 255 = 25
Byte 3: 50 AND 240 = ?
50 = 0011 0010
240 = 1111 0000
AND = 0011 0000 = 48
Byte 4: 22 AND 0 = 0
Network address: 172.25.48.0 ✓
⚠ Trap
Do NOT assume the third byte is unchanged. 50 AND 240 = 48, not 50. Always convert the non-trivial byte to binary and AND carefully.
Practice
IP = 10.45.130.77, Netmask = 255.255.128.0. What is the network address?
10 AND 255 = 10
45 AND 255 = 45
130 AND 128: 130=10000010, 128=10000000 → AND=10000000=128
77 AND 0 = 0
Network: 10.45.128.0

Longest prefix matching
Q25 type · 3 marks
▶
-- Algorithm -- 1. For each table entry, check if dst IP matches the prefix 2. A match means: IP AND mask == network_address 3. Among ALL matching entries, pick the LONGEST prefix 4. 0.0.0.0/0 (default) always matches — use ONLY if nothing else matches
Q25: dst=128.20.15.7, table: 128.0.0.0/8→P1, 128.20.0.0/16→P2, 128.20.15.0/24→P3, 0.0.0.0/0→P4
128.0.0.0/8: 128.20.15.7 starts with 128 ✓ (matches)
128.20.0.0/16: starts with 128.20 ✓ (matches)
128.20.15.0/24: starts with 128.20.15 ✓ (matches)
0.0.0.0/0: always matches ✓
Longest match = /24 → Port 3
Answer: Port 3 ✓
Practice
Destination IP: 192.168.5.200. Forwarding table: 192.0.0.0/8→P1, 192.168.0.0/16→P2, 192.168.5.0/24→P3, 192.168.5.192/26→P4. Which port?
All 4 entries match (192.168.5.200 is inside all those networks).
Longest match = /26. Check: 192.168.5.192/26 means .192–.255.
200 is in that range ✓ → Port 4

1
Buffer fill time
Q26 type · 3 marks
▶
net_fill_rate = send_rate − read_rate // KB/s time_to_fill = buffer_size / net_fill_rate // seconds
Q26: buffer=30KB, send=10KB/s, read=6KB/s. When does buffer fill?
net rate = 10 − 6 = 4 KB/s
time = 30 / 4 = 7.5 s
Answer: 7.50 s ✓ (range: 7.40–7.60)
⚠ Trap
Do NOT use the send rate alone. The receiver app drains the buffer simultaneously. Only the NET inflow rate (send minus read) fills the buffer.
Practice
Buffer = 50 KB. Sender writes at 15 KB/s, app reads at 8 KB/s. How many seconds until the buffer is full?
net = 15−8 = 7 KB/s
time = 50/7 = 7.14 s

Flow-control throughput from advertised window
Q27 type · 3 marks
▶
throughput = advertised_window / RTT -- Units -- window in KB: convert to bytes (×1024) first RTT in ms: convert to seconds (÷1000) first Result in B/s → KB/s (÷1024)
Q27: adv_window=32KB, RTT=80ms. Throughput in KB/s?
window = 32 × 1024 = 32,768 bytes
RTT = 80 ms = 0.080 s
throughput = 32,768 / 0.080 = 409,600 B/s
= 409,600 / 1024 ≈ 400 KB/s
Answer: ≈400 KB/s ✓ (range: 340–410)
Practice
TCP connection: advertised window = 64 KB, RTT = 200 ms. What is the max throughput in KB/s?
64×1024 = 65,536 B
65,536 / 0.2 = 327,680 B/s
/ 1024 = 320 KB/s

Dijkstra's Algorithm
W8 · Link state routing · Shortest path · Forwarding tables
1
Dijkstra step-by-step + forwarding table
Q36 type · 4 marks
▶
-- Initialise -- N' = {src} D(src) = 0 D(neighbour) = c(src, neighbour) D(non-neighbour) = ∞ -- Repeat until all nodes in N' -- 1. Pick w ∉ N' with minimum D(w) → add to N' 2. For each neighbour v of w not in N': D(v) = min(D(v), D(w) + c(w,v)) If updated: p(v) = w // predecessor
Exact Q2 PYQ — Q36 worked solution

Graph: U–V=3, U–W=6, U–X=2, X–Y=2, X–W=4, Y–W=3, Y–Z=1. Find shortest path cost from U to Z.

StepN'D(V)D(W)D(X)D(Y)D(Z)Pick
0{U}3,U6,U2,U∞∞—
1{U,X}3,Umin(6,2+4)=6,U—2+2=4,X∞X(D=2)
2{U,X,V}—6,U—4,X∞V(D=3)
3{U,X,V,Y}—min(6,4+3)=6,U——4+1=5,YY(D=4)
4{U,X,V,Y,Z}—6,U———Z(D=5) ✓
Shortest path U→Z = U→X→Y→Z, total cost = 2+2+1 = 5 ✓
To find forwarding table entry for Z: trace back p(Z)=Y, p(Y)=X, p(X)=U. First hop from U is X → forward to port connected to X.
Practice
Graph (undirected): A–B=4, A–C=2, B–D=3, C–B=1, C–D=5. Using Dijkstra from A, what is the shortest path cost from A to D?
Init: D(B)=4, D(C)=2, D(D)=∞
Step1: pick C(2). Relax: D(B)=min(4,2+1)=3,C. D(D)=min(∞,2+5)=7,C
Step2: pick B(3). Relax: D(D)=min(7,3+3)=6,B
Step3: pick D(6) ✓
Path: A→C→B→D, cost = 6
